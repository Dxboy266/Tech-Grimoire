# Java技术栈知识沉淀写作规范

> 将冗长技术文档拆分为易读模块

---

## 使用流程

1. **AI读文档** → 识别技术栈 → 匹配专家委员会
2. **以专家视角审核** → 给出拆分建议
3. **无需用户配置**

## 篇幅标准

| 类型 | 字数 | 阅读时长 |
|------|------|---------|
| 速查卡片 | 800-1500 | 3-5分钟 |
| 标准模块 | 2500-4000 | 10-15分钟 |
| 深度专题 | 5000-8000 | 20-30分钟 |

**拆分信号**: 超8000字 / 超3主题 / 超30分钟 / 超4层目录

---

## 审核标准

### 1. 技术准确性
- [ ] 原理是否正确？
- [ ] 是否有误导？
- [ ] 是否触及核心机制？

### 2. 拆分合理性
- [ ] 是否独立完整？
- [ ] 是否过度碎片化？
- [ ] 依赖关系是否清晰？

### 3. 生产可用性
- [ ] 有真实案例？
- [ ] 有性能数据？
- [ ] 考虑生产复杂性？

### 4. 可读性
- [ ] 代码可运行？
- [ ] 有图表辅助？
- [ ] 篇幅合理？

## 审核示例

**文档**: MySQL核心知识(15000字)

```
识别: MySQL - 审核专家: 丁奇, 姜承尧

技术准确性: ✅通过

拆分合理性: ❌需重新规划
当前: 单文档15000字，50分钟
建议拆分:
├── MySQL-架构与存储引擎.md (3500字)
├── MySQL-索引原理与优化.md (4000字)
├── MySQL-事务与锁机制.md (3800字)
└── MySQL-日志与主从复制.md (3700字)

生产可用性: ⚠️需补充
缺失: 慢查询优化完整案例

可读性: ⚠️需优化
问题: 目录5层，段落过长

丁奇: "内容质量高，但太长了。按建议拆分，每篇聚焦一个主题。"
```

## 文档结构模板

### 速查卡片
```
# 标题
> 一句话总结
## 核心概念
## 快速上手 (代码≤20行)
## 关键参数
## 常见坑点
```

### 标准模块
```
# 标题
## 📌 快速导航
## 一、快速上手
## 二、核心原理
## 三、生产实践
## 四、常见问题
```

### 深度专题
```
# 标题
## 📌 内容导航 (标注时长)
## 第一部分: 基础架构
## 第二部分: 核心机制
## 第三部分: 性能优化
## 第四部分: 生产实战
## 总结
```

## 质量红线

- ❌ 单文档超10000字
- ❌ 没有代码示例
- ❌ 代码无法运行
- ❌ 没有实战案例
- ❌ 目录超5层
